# Photoshop Controller (`photoshop-ctrl`)

## Project goal
A standalone hardware control surface for Adobe Photoshop, running on a Raspberry Pi 5
with a 7" DSI touchscreen and 8 rotary encoders. The Pi is the surface itself
(its own UI, its own touchscreen) and communicates with Photoshop on a separate
Mac/Windows host over the network.

The codebase was forked from a working Lightroom controller (`lightroom-ctrl`)
and is being adapted. Expect significant overlap in hardware abstraction, QML
chrome, and transport layers, but the host-comm protocol and action mappings
are Photoshop-specific and need to be replaced wholesale.

## Hardware
- Raspberry Pi 5, 4 GB
- Raspberry Pi 7" DSI touchscreen (800×480)
- 8 rotary encoders via two Adafruit **Quad Rotary Encoder** breakouts
  (I²C, seesaw firmware, 4 encoders + push-switches + NeoPixels per board)
- Active cooler, 5V/5A USB-C PSU

Pi 5 specifics worth remembering:
- GPIO lives on `gpiochip4` (not `gpiochip0` like older Pis)
- Use `libgpiod` v2 for GPIO; old wiringPi-style code does not apply
- Each Adafruit Quad board has 2 address-select jumpers → max 4 addresses
  on a single I²C bus → max 16 encoders without a TCA9548A multiplexer

## OS / runtime
- Raspberry Pi OS Lite, 64-bit (Bookworm); no desktop environment
- Qt app launched via systemd service at boot
- Display surface via Qt's **`eglfs`** platform plugin
  (`QT_QPA_PLATFORM=eglfs`); KMS/DRM directly, no X server, no Wayland compositor
- Touch input via evdev (auto-detected by eglfs)

## Build system
- **qmake**-based project (`photoshop-ctrl.pro`), not CMake — inherited from
  Lightroom controller; not worth porting right now
- Qt 6 (Bookworm apt repo ships 6.4; suitable for this project)
- Build natively on the Pi (no cross-compile)
- Standard build:
  ```bash
  qmake photoshop-ctrl.pro && make -j4
  ```
- IntelliSense via `compile_commands.json` generated by Bear:
  ```bash
  make clean
  qmake && bear -- make -j4
  ```

## Development workflow
- VS Code with Remote-SSH connected to the Pi
- C/C++ extension + clangd consume `compile_commands.json`
- Source on Pi at `~/projects/photoshop-ctrl`
- Fresh git repo, separate history from `lightroom-ctrl`
- `.gitignore` covers qmake artefacts (`build*/`, `Makefile*`, `moc_*`, `qrc_*`,
  `ui_*`, `.qmake.stash`, `*.pro.user*`, `compile_commands.json`)

## Architecture (three layers)

1. **Hardware-IO worker** — owns the I²C bus (`/dev/i2c-1`), reads the Adafruit
   seesaw boards (encoder deltas, button bitmask), debounces, drives the
   NeoPixels for visual feedback. Runs on its own `QThread`. Wire each board's
   INT pin to a Pi GPIO and use `libgpiod` + `QSocketNotifier` to wake on
   change rather than busy-poll. Emits Qt signals like
   `encoderTurned(int id, int delta)` and `buttonPressed(int id)`.

2. **Mapping/binding engine** — translates physical events into logical commands
   (`brush_size_delta`, `select_tool:brush`, `next_layer`, etc.). Driven by
   JSON/YAML profile files so the same encoder can mean brush size in paint
   mode and exposure in colour-grade mode.

3. **Network transport** — `QWebSocket` carrying JSON messages to/from the
   Photoshop host. Bidirectional: Photoshop sends current tool, active layer,
   document state back so the QML UI can show real context. Handles reconnect
   with backoff.

QML UI on the 7" panel: mode/profile indicator, on-screen controls that
complement the encoders, connection status, and (eventually) a live mapping editor.

## Photoshop-side integration
- **UXP plugin** (Adobe's current modern plugin platform; CEP/ExtendScript is
  sunsetting — do not start there)
- Plugin opens a WebSocket to the Pi, receives JSON commands, dispatches via
  `batchPlay` action descriptors and direct DOM APIs
  (`app.activeDocument`, `app.currentTool.brush`, etc.)
- Use the **Alchemist** plugin during development to record real Photoshop
  actions and copy out their `batchPlay` descriptor JSON — the UXP docs
  alone are not enough
- Plugin subscribes to Photoshop notifications (tool change, document change,
  layer change) and forwards them back over the WebSocket so the Pi UI stays
  in sync

## Why these choices

- **WebSocket over MIDI**: the Pi is a full surface with its own touchscreen,
  not a tethered MCU. We want bidirectional structured messages and rich
  Photoshop state pushed back, which MIDI's 7-bit world makes painful.
- **UXP over CEP**: Adobe's supported modern platform; CEP is on the way out.
- **eglfs over Wayland/X**: kiosk-style appliance — single full-screen Qt app,
  fastest boot, least overhead.
- **qmake over CMake**: inherited; rewriting the build system isn't on the
  critical path.

## Use cases prioritised
1. Brush & tool params — size, hardness, opacity, flow, tool switching
2. Adjustment-layer sliders — curves, levels, hue/sat, exposure
3. Layers & navigation — selection, visibility, zoom, undo/redo, history scrub

## Conventions
- C++17 minimum (Qt 6.4 supports it)
- Qt-style: `Q_OBJECT`, signals/slots, `QObject` parentage for ownership
- I²C reads run on a worker `QThread`; never block the GUI thread
- All host-comm messages are JSON; document the protocol schema in
  `docs/protocol.md` as it stabilises
- Match the existing code style inherited from `lightroom-ctrl`
